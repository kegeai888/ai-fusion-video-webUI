# 项目二次开发、优化与修复记录

## 1. 本次改造目标

本轮改造的核心目标是让项目在当前这类特殊容器环境中，通过 `/start.d` 下的可执行 shell 脚本，在开机后自动完成以下服务启动：

- 后端 API
- MySQL
- Redis
- 前端 Web（7860 端口）

约束条件：

- 必须使用 `.sh` 脚本
- 不使用 `systemd`
- 适配当前环境“把脚本放入 `/start.d` 并赋予可执行权限即可开机启动”的机制

---

## 2. 当前启动架构

### 2.1 推荐的一键启动入口

当前推荐的完整启动入口是：

```bash
bash /root/ai-fusion-video/boot-start.sh
```

这个脚本负责：

1. 检查 MySQL 是否可用
2. 检查 Redis 是否可用
3. 优先复用本机已有服务
4. 本机服务不可用时尝试拉起本地 service
5. 若仍不可用，再尝试通过 Docker Compose 拉起中间件
6. 将数据库和 Redis 连接信息注入后端启动环境
7. 调用 `start_app.sh` 拉起前后端
8. 对前端和后端进行启动后健康检查

因此：

- `boot-start.sh` 是“整套系统的一键启动脚本”
- `start_app.sh` 是“应用层启动脚本”，更偏向前后端本身的启动

### 2.2 各脚本职责划分

#### `boot-start.sh`
职责：

- 中间件兜底启动
- 环境变量准备
- 调用应用启动脚本
- 整体启动链路健康检查

#### `start_app.sh`
职责：

- 清理前端 7860 端口占用
- 清理后端 18080 端口占用
- 启动 Spring Boot 后端
- 启动 Next.js 前端
- 等待前后端完成就绪
- 输出日志路径与状态信息

#### `test_login.sh`
职责：

- 校验前端是否可访问
- 校验后端是否可访问
- 校验未登录时认证拦截是否正常
- 发起登录请求
- 校验 accessToken 是否可用
- 校验用户信息接口是否正常返回

## 3. 已完成的关键优化与修复

## 3.1 实现整套服务的 shell 启动链路

已完成从 `/start.d` 场景出发的 shell 启动方案设计与实现，满足特殊环境不依赖 `systemd` 的要求。

落地方式：

- 使用 `boot-start.sh` 作为完整启动入口
- 使用 `start_app.sh` 承担应用层拉起职责
- 脚本可复制到 `/start.d` 执行

经验：

- 这类环境下不要按传统 Linux 服务管理思路处理，而应尽量把所有启动依赖收敛到单一 shell 启动链中
- 一个主入口脚本 + 一个应用启动脚本，比拆成太多散乱脚本更利于维护

---

## 3.2 MySQL 启动与连接兜底逻辑完善

### 已实现逻辑

`boot-start.sh` 中的 MySQL 启动流程目前为：

1. 先检查远端/映射端口 `43306`
2. 如不可用，再检查本地 `3306`
3. 如本地未启动，则尝试 `service mysql start` 或 `service mysqld start`
4. 如仍不可用，再尝试 Docker Compose 拉起 mysql
5. 最终将正确的 `SPRING_DATASOURCE_URL` 注入后端

### 收益

- 提高不同部署环境下的兼容性
- 避免因为只写死一种数据库来源，导致后端启动失败

经验：

- 启动脚本不要假设数据库来源唯一，应该优先“能复用就复用，不能复用再创建”
- 数据库 readiness 检查不能只看端口，需要结合 `mysqladmin ping`

---

## 3.3 Redis 启动与连接兜底逻辑完善

### 已实现逻辑

`boot-start.sh` 中 Redis 处理流程与 MySQL 类似：

1. 先检查映射端口 `46379`
2. 再检查本地 `6379`
3. 本地未启动时尝试 `service redis-server start` 或 `service redis start`
4. 如仍不可用，再尝试 Docker Compose 拉起 redis
5. 将 `SPRING_DATA_REDIS_HOST` 与 `SPRING_DATA_REDIS_PORT` 注入后端环境

### 收益

- 降低 Redis 未预先启动导致的后端失败概率
- 提高在本地服务 / Docker 中间件两种运行模式下的兼容性

经验：

- Redis 检查比单纯探测端口更稳妥的方式是 `redis-cli ping`
- 启动脚本应先判断已有实例是否可用，而不是无条件重建实例

## 3.4 修复后端因空 Redis 环境变量导致的启动失败

### 问题现象

后端曾出现如下问题：

- `SPRING_DATA_REDIS_PORT` 被传入空字符串
- Spring Boot 绑定 `spring.data.redis.port` 到 `int` 时失败
- 后端启动中断

### 根因

`start_app.sh` 早期写法会在变量为空时仍然传入：

- `SPRING_DATA_REDIS_PORT=""`

这会让 Spring 误以为用户显式配置了一个空值。

### 修复方式

引入 `build_backend_env()`，仅在变量非空时才把它们加入后端进程环境。

### 结果

这个问题已经消除，后端可以正常读取 Redis 配置。

经验：

- 给 Spring Boot 传环境变量时，“空字符串”和“未设置”不是一回事
- 对可选环境变量，脚本层应当做到“仅在有值时才注入”

## 3.5 修复前端登录验证脚本中的接口路径错误

### 问题现象

登录验证脚本 `test_login.sh` 早期使用了错误路径：

- `/api/auth/login`
- `/api/auth/user-info`

导致脚本校验失败，与真实后端接口不一致。

### 根因

后端真实控制器路由是：

- `/auth/login`
- `/auth/user-info`

不是 `/api/auth/...`

### 修复方式

已修正 `test_login.sh` 中接口路径，使其与后端实际控制器保持一致。

### 结果

`test_login.sh` 现在可用于完整验证：

- 前端可达
- 后端可达
- 认证拦截正常
- 登录成功
- Token 有效
- 用户信息接口返回正常

经验：

- 不要凭前端代理路径去猜测后端控制器实际路由
- 验证脚本必须以真实后端 Controller 映射为准

## 3.6 修复后端健康检查路径不稳定的问题

### 问题现象

早期启动检查曾直接使用：

```bash
http://127.0.0.1:18080/swagger-ui.html
```

但在当前运行态下，这个路径并不稳定，曾出现 404。

### 根因

项目实际运行时存在 `/api` context path，导致文档接口的稳定可探测路径更适合使用：

```bash
http://127.0.0.1:18080/api/v3/api-docs
```

### 修复方式

已在 `boot-start.sh` 中将后端启动后检查切换到更稳定的 `api-docs` 接口。

### 结果

后端 readiness 判断更可靠，不再因 swagger 页面路径差异出现误判。

经验：

- 健康检查优先选择“稳定、轻量、返回码明确”的接口
- 不要把 UI 页面地址当成唯一健康检查依据

## 3.7 增强前端 7860 端口清理能力

### 问题现象

前端多次出现：

```text
Error: listen EADDRINUSE: address already in use 0.0.0.0:7860
```

即 7860 端口被旧的 Next.js 进程占用，导致新前端进程无法启动。

### 已做优化

`start_app.sh` 已增加：

- 基于 `lsof` 的端口占用检查
- 基于进程参数的 Next.js 进程识别
- 对前端相关进程执行清理
- 必要时递归清理进程树
- 启动后检查新前端进程是否提前退出
- 等待端口释放后再继续启动

### 当前结论

相比初版已经明显增强，但这个问题在复杂残留进程场景下仍然有一定波动。

也就是说：

- 脚本层已具备较好的自动清理能力
- 但某些遗留 Next.js 进程树场景下，仍可能需要人工补充清理

经验：

- 端口冲突问题不能只 kill 一个父进程，往往要考虑整棵进程树
- 判断“前端是否启动成功”不能只看端口通不通，还要看新进程是否存活
- 在开发态 `pnpm dev` / `next dev` 环境下，孤儿进程比生产态更常见

## 3.8 修复前端“假启动成功”的问题

### 问题现象

曾出现这种情况：

- 旧前端进程仍占着 7860
- 新前端进程其实启动失败
- 但脚本如果只看端口可访问，会误以为启动成功

### 修复方式

在 `start_app.sh` 中增加了新前端进程的早期存活检查：

- 启动后持续检查 `FRONTEND_PID`
- 如果前几秒内就退出，则立即判定启动失败
- 提示查看 `frontend.log`

### 结果

减少了“误判成功”的情况，问题定位更直接。

经验：

- 启动脚本应区分“端口上有人”和“我刚启动的进程真的活着”这两件事
- 进程存活检查是开发态服务启动脚本里非常关键的一层保护

## 5. 当前已知问题与后续建议

## 5.1 已知问题：7860 端口清理仍可继续增强

虽然目前脚本已经比初版稳定很多，但前端 `7860` 端口占用问题仍是当前启动链中最值得继续优化的点。

主要原因：

- Next.js 开发态进程树较复杂
- 历史残留子进程可能绕过单层 kill 逻辑
- 某些情况下旧监听还在，导致新实例 `EADDRINUSE`

后续建议：

1. 继续增强端口占用进程识别逻辑
2. 优先按端口关联 PID 做整棵树清理
3. 必要时记录被清理的 PID/命令行到日志，方便排查
4. 如果后续转生产部署，可考虑用 `next start` 或静态部署，减少 dev server 残留问题

## 6. 本轮改造的核心经验总结

### 经验 1：特殊容器环境下，shell 启动链比传统服务编排更重要

在当前环境里，最关键的不是 system service 管理，而是：

- 启动顺序
- 依赖可用性检查
- 失败时的兜底策略
- 日志可追踪性

### 经验 2：启动成功不等于服务真正可用

必须区分：

- 进程启动了
- 端口监听了
- 接口真的可访问
- 业务链路真的可用

因此需要：

- 端口检查
- HTTP 健康检查
- 登录链路回归测试

### 经验 3：脚本里的“空环境变量”很容易引发框架级错误

对于 Spring Boot 这类配置绑定框架，脚本层注入变量时必须很谨慎。

原则：

- 有值才传
- 没值就不要传

### 经验 4：开发态前端进程最容易留下孤儿进程

尤其是 `pnpm dev` / `next dev` 这种模式，端口冲突和残留子进程问题远比生产态常见。

这类问题需要：

- 查 PID
- 查父子进程关系
- 必要时杀进程树
- 启动后再校验新进程是否仍存活

### 经验 5：健康检查接口要选稳定路径

不要把页面地址、跳转地址、依赖 context path 的地址，直接当作唯一探针。

优先选择：

- 返回值稳定
- 语义明确
- 不依赖页面渲染
- 不容易受鉴权和重定向干扰

### 经验 6：联调测试脚本非常值得保留

`test_login.sh` 这类脚本可以大幅降低后续排查成本。

后续任何涉及以下内容的改动，都应该保留并复用这类脚本：

- 认证
- 路由
- API 前缀
- 代理配置
- 启动脚本
